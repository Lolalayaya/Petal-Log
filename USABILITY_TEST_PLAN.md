---

## 1. 測試目標

1. 驗證新手使用者能否**不靠任何說明**，獨立完成「記錄一筆經期」與「看懂週期預測」這兩項核心任務。
2. 找出「進階功能」的**可發現性（discoverability）**問題——週期階段顏色、症狀追蹤、自訂症狀、異常提醒、報表匯出這幾項功能預設是**收合或關閉**的，需要驗證使用者是否能自己找到、是否理解用途。
3. 檢查文案與資訊架構是否會造成理解落差，例如：
   - 週期天數卡片在「本次週期第 X 天」與「距離下次還有 X 天」之間切換的邏輯，使用者能否理解為什麼字變了？
   - 易孕期／排卵日的免責提示是否被閱讀、是否被誤解為醫療建議？
   - 異常提醒的判斷標準（經期過長／週期不規律）是否清楚？
4. 蒐集使用者對隱私設計（無帳號、中性文字、純本機儲存、可列印 PDF 報表）的觀感與信任程度。
5. （作品集用）產出可量化的任務完成率、卡關次數、滿意度數據，作為設計決策的佐證。

---

## 2. 測試對象

- **人數建議**：5–8 位。可用性測試的經驗法則是 5 位就能找出約 80% 的重大問題，但這款 App 涵蓋「有記錄習慣」與「沒用過經期 App」兩種使用者，建議至少各找 2–3 位混合測試，比較經驗落差。
- **招募條件**：會用手機或電腦瀏覽器操作即可，不需要技術背景。
- **建議排除**：已經深度參與開發過程、看過操作方式或截圖的人（觀察會失真）。
- **可選分組**：若人數足夠，可分「曾用過其他經期 App」vs「從未用過」兩組，比較心智模型差異（例如對「易孕期」「安全期」既有認知是否跟本 App 的呈現方式衝突）。

## 3. 測試前準備

- **裝置**：優先用手機瀏覽器測試（App 是 mobile-first 設計，`App.module.css` 的 `.app` 寬度上限 480px）；若條件允許，額外測一次桌機瀏覽器，確認體驗一致。
- **網址**：本機 `npm run dev`（見 README）或部署後的 GitHub Pages 網址。
- **資料狀態**：每位測試者開始前，務必**用無痕視窗**，或手動清除 `localStorage`（設定 → 清除所有本機紀錄），確保每個人都是「全新使用者」視角，避免看到別人測試留下的資料。
- **錄影/錄音**：徵求測試者同意後錄螢幕操作過程；若不方便錄影，至少安排一位觀察者專門記筆記，主持人專心引導、不邊做筆記邊主持。
- **計時工具**：碼表或手機計時，記錄每個任務花費時間。
- **列印本文件第 8 章的觀察記錄表**，方便手寫紀錄。

## 4. 開場腳本（暖場）

主持人可參考以下說法（不必逐字念，但涵蓋重點）：

> 「謝謝你來幫忙測試。這是一個經期記錄的網頁應用，接下來我會請你完成幾個任務。**這不是在測試你，是在測試這個產品**——如果哪個地方卡住了、找不到，那是產品設計的問題，不是你的問題，請不用有壓力。
>
> 操作的時候麻煩你**盡量把心裡想的講出來**（think-aloud），比如「我現在想找設定」「我不確定這個按鈕是做什麼的」，這樣我才能知道你的思考過程。
>
> 過程中我可能不會直接回答你的問題，會反問你「你覺得這是做什麼的？」，這是測試方法的一部分，不是故意不理你，請見諒。
>
> 這是一個記錄經期的 App，如果你覺得這個主題讓你不自在，我們可以隨時暫停或調整任務內容。」

**務必詢問並記錄**：是否同意錄影/錄音、是否對此主題感到自在。

## 5. 任務清單

每個任務包含：**情境提示**（唸給測試者聽的話）、**預期完成路徑**、**觀察重點**、**已知風險點**（主持人心裡有數，但不要暗示測試者）。任務之間**不要主動提示**還沒發現的功能，除非測試者完全卡住超過 2 分鐘。

### Task 1：初次開啟與引導

> 「請打開這個網址，把你看到的畫面當作第一次使用，照你覺得對的方式操作下去。」

- **預期路徑**：看到「歡迎使用 Petal Log」→ 選擇「有，我想輸入」或「沒有，先用預設值」→（若選有）輸入平均經期/週期天數 → 完成設定。
- **觀察重點**：
  - 使用者能否理解「平均經期天數」與「大約幾天來一次」兩個問題在問什麼？
  - 有沒有人猶豫要選「有」還是「沒有」？猶豫的原因是什麼？
  - 完成後，使用者是否理解自己已經進入主畫面？
- **已知風險點**：兩個輸入框的說明文字（「大約幾天來一次（這次到下次第一天）」）括號補充是否多餘或不夠清楚。

### Task 2：記錄今天的經期

> 「假設你今天經期來了，請把它記錄下來。」

- **預期路徑**：點主畫面下方的「+ 記錄」FAB → 選經量 → 儲存。
- **觀察重點**：
  - 使用者能否在 10 秒內找到記錄入口？（對應 SDD 的「10 秒完成一筆記錄」目標）
  - 「量少/量中/量多」的選擇是否直覺？
  - 儲存後，使用者是否注意到月曆上出現了標記、上方出現了預測卡片？
- **已知風險點**：FAB 上只寫「記錄」兩個字（中性文字設計），初次見到的使用者是否能立刻理解點下去會發生什麼事，還是需要試了才知道？

### Task 3：看懂週期預測資訊

> 「現在請看一下畫面最上方的資訊卡片，跟我說說你從這幾張卡片看懂了什麼。」

- **預期路徑**：口頭描述 + 指認「本次週期第 X 天」「預計下次」「易孕期」三張卡片的意思。
- **觀察重點**：
  - 使用者是否正確理解「易孕期」卡片下方那行小字（免責提示）？會不會誤以為是精準醫療數據？
  - 若目前不在經期中（Task 2 若選在較早之前的日期，此卡片會顯示「距離下次」而非「本次週期」），使用者能否理解為什麼寫法變了？
- **已知風險點**：這是特別想驗證的一個邏輯——「本次週期第幾天」與「距離下次」的動態切換，是為了避免經期結束後還顯示令人困惑的天數，但這個「聰明」的設計對第一次看到的使用者來說，可能反而更難懂。

### Task 4：補記錄或修改過去的日期

> 「假設你發現上禮拜有一天忘記記錄了，請幫我補上去；然後再假設你記錯經量了，請把它改掉。」

- **預期路徑**：點月曆上的某天 → 開啟日詳情 → 選經量 → 儲存；再點同一天 → 修改經量 → 更新。
- **觀察重點**：
  - 使用者能否分辨「已記錄的日期」（玫瑰色）和「預測日期」（虛線）的視覺差異？
  - 修改與刪除的操作是否容易找到？
- **已知風險點**：無特別已知問題，此任務主要作為對照組，確認核心 CRUD 流程沒有退化。

### Task 5：找到症狀記錄功能

> 「如果你這次經期有一些不舒服的症狀想記錄下來，比如頭痛或經痛，請試著找找看這個 App 能不能做到。」

- **預期路徑**：進設定 → 開啟「記錄伴隨症狀」開關 → 回到日曆點一天 → 看到症狀選單。
- **觀察重點**：**這是本次測試最重要的觀察項目之一。**
  - 使用者第一直覺會去哪裡找？（設定？日詳情？FAB？）
  - 找到「記錄伴隨症狀」開關後，是否理解打開後要去哪裡用？
  - 完成任務花費的時間、中途放棄或求助的次數。
- **已知風險點**：症狀記錄預設是**關閉**的，且開關本身不會主動提示「開啟後請到日詳情使用」，使用者很可能打開開關後不知道下一步該做什麼。這是最值得驗證、也最可能發現真實問題的任務。

### Task 6：記錄自訂症狀與備註

> 「請在剛剛那個症狀選單裡，新增一個 App 沒有提供、但你想記錄的症狀；然後再用『其他』欄位寫一段文字備註。」

- **預期路徑**：日詳情的症狀選單 → 點「其他」晶片 → 出現文字框 → 輸入文字；另外進設定展開「症狀項目與顏色設定」手風琴 → 新增症狀名稱 → 回到日詳情確認新症狀出現。
- **觀察重點**：
  - 使用者是否理解「其他」是「這次臨時想寫的話」，跟「新增自訂症狀」（設定裡，永久新增一個症狀選項）是兩件不同的事？這兩個功能容易混淆。
  - 手風琴收合的「症狀項目與顏色設定」是否容易被發現？
- **已知風險點**：「其他」（單次備註）與「自訂症狀」（永久選項）概念相近但機制不同，是本次最想驗證的概念混淆風險。

### Task 7：找到週期階段顏色設定

> 「這個 App 其實可以在月曆上用不同顏色標出『月經期、濾泡期、排卵期、黃體期』四個生理階段，請試著把這個功能打開。」

- **預期路徑**：進設定 → 找到「週期階段顏色提示」手風琴 → 展開 → 打開任一階段的開關。
- **觀察重點**：
  - 使用者是否知道要進「設定」找？
  - 展開手風琴後，是否理解「色票」跟「開關」分別在控制什麼（顏色 vs 是否顯示）？
  - 打開後回到月曆，是否有注意到日期格子上方出現的色條？（色條很細，容易被忽略）
- **已知風險點**：色條只有 3px 高，且位置在日期數字上方，视覺上可能不夠顯眼；此外「濾泡期」「黃體期」等醫學名詞對非醫療背景使用者是否有理解障礙，值得口頭確認。

### Task 8：異常提醒與報表匯出（情境模擬任務）

> 「現在請想像一個情境：這個 App 跳出提醒，說你的週期不太規律。請你照著畫面的引導，試著把相關資料整理成一份可以拿給醫生看的文件。」

執行前，主持人先在後台（或請測試者用瀏覽器開發工具）植入幾筆會觸發異常提醒的資料，或直接準備一台**已經有異常資料**的測試機給這個任務用（比較不打斷測試者自己建立的資料脈絡）。

- **預期路徑**：看到主畫面的異常提醒橫幅 → 點「查看報表」→ 瀏覽報表內容 → 點「列印／另存為 PDF」。
- **觀察重點**：
  - 使用者是否讀懂異常提醒的文字（次數＋原因）？
  - 是否理解「查看報表」會離開目前畫面、進入一個新頁面？
  - 報表裡的「摘要統計」「週期歷史」「異常提醒」三個區塊，使用者最先看哪一個？
  - 使用者是否知道要用瀏覽器的列印功能才能存成 PDF（而不是找 App 裡有沒有「匯出」按鈕）？
- **已知風險點**：「列印／另存為 PDF」這顆按鈕的措辭是否讓使用者理解要透過瀏覽器原生列印對話框操作，而不是 App 直接生成檔案下載。

### Task 9：收尾與整體觀感

> 「測試就到這邊，最後想請你自由逛一下這個 App，看看還有沒有你覺得有趣、但我們剛剛沒有提到的功能。」

- **觀察重點**：使用者是否自己發現了前面任務沒引導到的功能（例如自動填滿開關、匯出報表以外的設定項）？這能反映整體資訊架構的可探索性。

## 6. 測後訪談問題

任務結束後，卸下「操作任務」的壓力，用聊天的方式問：

1. 整個過程中，哪一步讓你覺得**最順手**？為什麼？
2. 哪一步讓你**卡住或猶豫**？如果請你形容那個當下的感覺，會怎麼說？
3. 用一句話，你會怎麼跟朋友介紹這個 App？
4. 如果這是你自己要用的經期記錄工具，你會不會想要一個帳號登入、資料同步到雲端？為什麼？（用來驗證「純本地、零帳號」的定位是否符合使用者期待）
5. 剛剛設定裡的「異常提醒」「報表匯出」這類功能，你會不會實際去用？在什麼情況下你會想用？
6. 你會推薦這個 App 給朋友嗎？0–10 分，你會打幾分？為什麼？（Net Promoter Score 簡化版）

**選用：SUS（System Usability Scale）簡化版**，可從標準 10 題挑 5 題快速施測，1（非常不同意）到 5（非常同意）：

1. 我覺得這個 App 很容易上手。
2. 我覺得這個 App 的功能太多、太複雜。
3. 我覺得整個 App 用起來很一致（不會這裡一種邏輯、那裡又一種邏輯）。
4. 我需要別人教我才會用某些功能。
5. 我會想常常使用這個 App。

### 7.2 問題嚴重度分級

整理發現的問題時，建議用以下分級（常見的可用性測試慣例），方便排優先順序：

- **P0（阻斷）**：使用者完全無法完成任務，需要主持人介入才能繼續。
- **P1（造成困惑）**：使用者最終完成了，但過程中明顯猶豫、走錯路、或誤解了功能用途。
- **P2（錦上添花）**：使用者順利完成，但口頭表達「如果⋯⋯會更好」的建議。

### 7.3 案例研究寫作結構建議

若要整理成作品集頁面，建議用「發現 → 洞察 → 改善建議」的三段式呈現每個重大問題，例如：

> **發現**：8 位測試者中有 6 位在 Task 5（找到症狀記錄功能）花了超過 1 分鐘，其中 3 位中途需要提示才找到設定裡的開關。
>
> **洞察**：症狀追蹤功能為了不干擾核心的「快速記錄」流程，刻意設計成預設關閉、藏在設定裡，但這個設計選擇犧牲了可發現性，使用者的心智模型是「症狀應該跟記錄經量放在一起」，而不是「先去設定開啟功能」。
>
> **改善建議**：可以在日詳情面板裡，即使症狀追蹤是關閉狀態，也放一行低調的引導文字（例如「想記錄症狀嗎？到設定開啟」），或是把開關的入口從設定移到日詳情面板本身。
